## 🟡 推荐改进（有明确正收益）

### 1. Zustand `partialize` 将函数序列化到 localStorage 【中优先级·性能/存储】

**现状**：[store.ts L308-310](file:///i:/markdown2view/src/lib/store.ts#L308-L310) 的 `partialize` 只排除了 `guideTrigger`，但 store 里有大量 action 函数（`setArticleMarkdown`、`setTheme` 等十几个）。Zustand persist 默认会尝试序列化整个返回值——函数会被忽略不报错，但**每次 `set()` 都会触发 JSON.stringify 处理包含这些函数引用的对象**。

**建议**：`partialize` 应该只保留需要持久化的数据字段，显式排除所有 action 函数。这是 Zustand persist 的标准最佳实践。

```ts
partialize: (state) => ({
  articleMarkdown: state.articleMarkdown,
  documentMarkdown: state.documentMarkdown,
  cardMarkdown: state.cardMarkdown,
  html: state.html,
  mode: state.mode,
  inputType: state.inputType,
  platform: state.platform,
  documentSettings: state.documentSettings,
  articleFont: state.articleFont,
  cardFont: state.cardFont,
  accent: state.accent,
  accentDark: state.accentDark,
  colors: state.colors,
  imageHostConfig: state.imageHostConfig,
  demoVersion: state.demoVersion,
  articleDirty: state.articleDirty,
  documentDirty: state.documentDirty,
  cardDirty: state.cardDirty,
  htmlDirty: state.htmlDirty,
  customInstructions: state.customInstructions,
}),
```

**正收益**：减少每次 `set()` 时 JSON.stringify 处理的对象体积；代码意图更清晰（明确哪些字段持久化）。

### 2. App.tsx 过大（508 行），头部 Header + 移动端抽屉应拆分 【中优先级·可维护性】

**现状**：[App.tsx](file:///i:/markdown2view/src/App.tsx) 包含完整的 header 响应式逻辑（960/1300/1450px 断点）、移动端侧边抽屉（~150 行 JSX）、主题色选择器等，混在根组件中。

**建议**：把 header（L122-303）和移动端抽屉（L352-503）提取为独立组件 `<AppHeader>` 和 `<MobileDrawer>`。好处：
- App.tsx 缩减到 ~150 行，只关注模式路由和全局 Provider
- header 和 drawer 可独立维护和测试
- 利于未来国际化（字符串集中管理）

**注意**：这不是"为了拆而拆"——508 行的根组件、3 种响应式断点的分支逻辑、重复的 SVG 图标，已经到了影响阅读效率的程度。

### 3. 缺少 Error Boundary 【中优先级·健壮性】

**现状**：项目没有任何 Error Boundary。Markdown 引擎（835 行自定义解析器）、iframe 沙箱、KaTeX 渲染、图片导出等操作都可能抛出运行时异常。一旦未捕获，整个 UI 白屏。

**建议**：至少在以下位置添加 Error Boundary：
1. **`<Suspense>` 外层**包一个全局 Error Boundary（懒加载模块加载失败时兜底）
2. **预览面板**（引擎渲染异常不应影响编辑器）

推荐使用 React 的 class component Error Boundary（这是 React 唯一不支持 hook 的场景），约 30 行代码：

```tsx
class ErrorBoundary extends React.Component<{ fallback: ReactNode; children: ReactNode }, { hasError: boolean }> {
  state = { hasError: false }
  static getDerivedStateFromError() { return { hasError: true } }
  componentDidCatch(error: Error) { console.error('ErrorBoundary caught:', error) }
  render() { return this.state.hasError ? this.props.fallback : this.props.children }
}
```

**正收益**：防止渲染引擎 bug 导致全屏白屏；给用户提供"重试"能力。

### 4. `as any` 使用可收紧 【低优先级·类型安全】

**现状**：`src/` 中有 15 处 `as any`。大部分在测试文件（mock IndexedDB），可以接受。但有 2 处值得修：

| 位置 | 问题 | 建议 |
|---|---|---|
| [App.tsx L391](file:///i:/markdown2view/src/App.tsx#L391) | `setMode(m.key as any)` | 把 `m.key` 的数组声明为 `{ key: RenderMode; ... }[]`，即 `as const` 或显式类型 |
| [store.ts L316](file:///i:/markdown2view/src/lib/store.ts#L316) | `(state as any).customInstructions = []` | 在 `onRehydrateStorage` 的回调参数类型中处理可选字段，或用 `state.customInstructions ??= []` |

**正收益**：编译器能捕获 typo（如 `m.key = 'artcle'`）。

---

### 5. 移动端抽屉按钮 CSS class 重复 【低优先级·DRY】

**现状**：[App.tsx](file:///i:/markdown2view/src/App.tsx#L412-L454) 中移动端抽屉的 4 个功能按钮（帮助/设置/恢复/隐私）使用完全相同的 className：
```
"w-full flex items-center gap-3 px-4 py-3 rounded-xl border border-slate-200 bg-slate-50/50 hover:bg-slate-50 text-[13px] font-medium text-slate-700 transition-colors cursor-pointer"
```

**建议**：提取为一个 `<DrawerButton>` 组件或 CSS class。

## 🔵 暂不建议改动的

以下是研究中发现的点，但**在当前阶段收益不足以证明改动成本**：

| 观察 | 理由 |
|---|---|
| 没有 ESLint/Prettier | 个人/小团队项目、TypeScript 严格模式已覆盖大部分场景。如果未来开放社区贡献再加不迟 |
| 测试覆盖率低（仅 6 个 test 文件） | 核心解析器 and 模型逻辑有测试；UI 组件测试 ROI 低 |
| 无障碍（ARIA）不足 | 工具类应用、非面向公众的无障碍合规场景；Headless UI 自带基础 ARIA |
| `markdownParser.ts` 835 行 | 自定义 Markdown 解析器，逻辑耦合紧密，拆分反而增加跳转成本 |
| 各 Mode 组件 400-700 行 | 每个 mode 本身是独立场景，内部逻辑自洽，不影响其他 mode |
| 主题色 SVG 图标内联 | 数量不多（~10 个），独立图标组件文件的维护成本 > 收益 |
| `fflate`、`jspdf` 未动态导入 | Vite 的 tree-shaking 已处理，且这些只在导出流程使用，lazy import 收益边际递减 |
