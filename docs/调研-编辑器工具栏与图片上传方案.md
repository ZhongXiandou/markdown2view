# 编辑器工具栏与图片上传方案调研

## 一、编辑器工具栏方案调研

### 1.1 需求概述

在现有 CodeMirror 6 编辑器上方添加工具栏，支持：
- 基础格式：加粗、斜体、删除线、行内代码
- 列表：有序列表、无序列表、任务列表
- 标题：H1-H4
- 图片上传
- 项目特色标签/内联样式（如 `<steps>`、`<lead>`、`<breaking>` 等自定义组件）

### 1.2 方案对比

#### 方案 A：自建工具栏 + CodeMirror 6 命令（推荐）

**思路**：在 CodeEditor 组件上方渲染一个 React 工具栏组件，点击按钮时通过 CodeMirror 6 的 `view.dispatch()` API 在光标位置插入/包裹 Markdown 语法。

**优势**：
- 零额外依赖，与现有架构完全一致
- 可完全自定义按钮和样式，包括项目特色标签
- 体积最小
- 按钮行为完全可控

**劣势**：
- 需要自行实现每个格式的插入与 Toggle 逻辑（约 200-300 行代码）

**实现复杂度**：低-中（1-2 天）

**参考实现**：
```tsx
// 核心：通过 view.dispatch 在光标位置插入文本
function insertAtCursor(view: EditorView, before: string, after = '') {
  const { from, to } = view.state.selection.main
  const selected = view.state.sliceDoc(from, to)
  view.dispatch({
    changes: { from, to, insert: `${before}${selected}${after}` },
    selection: { anchor: from + before.length, head: from + before.length + selected.length },
  })
  view.focus()
}
```

#### 方案 B：Milkdown（ProseMirror-based WYSIWYG）
**结论**：**不推荐**。底层从 CodeMirror 换成 ProseMirror，架构冲突太大，迁移成本远超收益。

#### 方案 C：ByteMD（已停止活跃维护）
**结论**：**不推荐**。底层基于 CodeMirror 5，版本过旧且不活跃。

#### 方案 D：md-editor-rt
**结论**：**可考虑但不优先**。虽然基于 CodeMirror 6，但需要整体替换现有编辑器，且自定义标签适配困难，包体积较大。

### 1.3 推荐方案

**方案 A：自建工具栏**，理由：
1. **架构零侵入**：只需在 CodeEditor 上方加一个 React 组件，不改变现有编辑器逻辑。
2. **项目特色标签**：可以直接定义 `<steps>`、`<lead>` 等按钮，其他通用编辑器方案都需要额外适配。
3. **快捷键原生集成**：可在 CodeMirror 扩展中通过 `keymap.of` 绑定快捷键（如 `Ctrl+B`、`Ctrl+I`），调用与工具栏按钮相同的逻辑。
4. **格式 Toggle 功能**：通过解析当前选区/光标的前后字符，支持对格式的智能反包裹（Unwrap），避免格式嵌套堆叠。

## 二、图片上传与存储方案调研（更新版）

本系统为**纯前端、零后端**架构，但对于微信公众号等场景，图片存储和引入方案直接决定了用户排版的成败。

### 2.1 存储机制选择：混合 IndexedDB 模式（最优解）

原方案提议“直接 Canvas 压缩后 base64 内联写入 Markdown”。经过评估，该方案存在 **localStorage 爆限 (5MB 瓶颈)** 和 **CodeMirror 编辑长 base64 文本严重卡顿** 的致命硬伤。

我们决定采用以下**混合存储与动态编译**模式：

```
[拖拽/粘贴图片] 
      │
      ▼
 [Canvas 压缩] ──► 得到二进制 Blob
      │
      ├───────────────────────────────┐
      ▼ (本地存储)                    ▼ (云图床已配置)
[写入 IndexedDB]               [上传至云图床]
      │                               │
      ▼                               ▼
[生成本地 img://id 占位]        [直接生成公网 HTTPS URL]
      │                               │
      ▼                               ▼
[写入 Markdown 源码]           [写入 Markdown 源码]
      │                               │
      ├───────────────────────────────┘
      ▼
 [渲染/导出处理]
```

#### 混合存储机制详解
1. **本地存储**：图片压缩后，存入 IndexedDB（拥有数百 MB 的大容量限额），生成唯一的引用 ID（如 `img_171810594328`）。Markdown 源码中仅记录 `![描述](img://img_171810594328)`。
2. **预览渲染**：
   - 应用启动时或解析到 `img://` 占位符时，从 IndexedDB 读取图片 Blob，并调用 `URL.createObjectURL(blob)` 映射为浏览器内存 URL。
   - `inlineFormat` 渲染引擎会将 `img://img_xxx` 动态替换为上述内存 URL，保证极速且流畅的预览。
   - **优势**：Markdown 源码极其轻量，`localStorage` 完美避免超限，CodeMirror 运行无延迟。
3. **复制/导出 (自包含编译)**：
   - 当用户点击“复制 Markdown”或“复制富文本”时，后台异步读取 IndexedDB 对应的图片数据，将其编译为自包含的 `base64` 编码，合并回最终文本再写入剪贴板。

### 2.2 微信公众号等外部平台的兼容性与第三方图床

#### 2.2.1 裂图痛点
微信公众号编辑器**不支持 base64 内联图片**。如果用户将本地 IndexedDB / base64 渲染出来的文章直接复制到微信编辑器中，微信后台会拦截并显示裂图。微信只支持公网 HTTP(S) 图片，或者用户手动在微信后台上传。

#### 2.2.2 解决方案
1. **公众号复制警告横幅 (WeChat Copy Warning)**：
   在文章（公众号）渲染预览区域，系统会自动检测 Markdown 源码。若检测到含有 `img://` 本地占位符，将在顶部合适位置展示一个精美的警示横幅：
   > ⚠️ **温馨提示**：当前文章包含本地存储的图片。直接复制到微信公众号会导致图片失效（裂图）。建议配置第三方图床以自动上传，或手动在微信后台重新上传这些图片。
2. **第三方云存储与图床集成**：
   在系统设置中加入图片上传源配置。用户可按需选用以下服务（均为前端直接流式上传，不经过中转后端）：
   * **免费图床**：优先支持 Sm.ms API，或通过 GitHub 仓库 + jsDelivr CDN 搭建的免费图床。
   * **付费/企业云存储**：支持阿里云 OSS、腾讯云 COS、七牛云、MinIO 等（需用户输入 AccessKey、Bucket 等必要的前端配置信息）。
   * **逻辑路由**：若已配置图床，上传图片时直传云端并返回公网链接插入 Markdown；未配置时，自动降级为本地 IndexedDB 混合存储，用于本地预览、PDF 导出或单机工作。

## 三、 实现计划（第一阶段 MVP 修正版）

### 3.1 工具栏组件 (`src/components/editor/EditorToolbar.tsx`)
- 布局规划：
  ```
  [H1] [H2] [H3] | [B] [I] [Code] | [UL] [OL] [Task] | [Image] | [> Quote] [--- HR] | [步骤] [引言] [对比]
  ```
- 核心功能：
  - 点击按钮触发加粗、斜体等。
  - 支持**无选区时光标居中**及**选中时的格式 Toggle 反包裹**。
  - 支持多行选中时逐行插入列表前缀。
  - 块级标签（如 `<steps>`）在前后自动生成换行符。

### 3.2 图像处理器与存储引擎 (`src/lib/editor/imageStorage.ts`)
- **IndexedDB 封装**：简易数据库写入与读取，管理 `img://id` 到二进制 Blob 的持久化。
- **内存 URL 映射表**：管理 `Record<string, string>`（`img://id` -> `blob:url`）的生命周期。
- **Canvas 压缩器**：控制图片的最大宽度为 1600px，JPEG 质量为 0.7（公众号最优压缩参数），体积控制在 50-200KB。
- **图床直传服务**：封装免费图床（Sm.ms）与主流付费云存储的纯前端上传 SDK。

### 3.3 编辑器事件集成与快捷键
- 在 `CodeEditor.tsx` 中使用 `EditorView.domEventHandlers` 绑定 `paste` 与 `drop`。
- 在 `drop` 事件中，使用 `view.posAtCoords` 在光标拖放精确位置插入图片。
- 绑定快捷键 keymaps：`Ctrl+B` (加粗), `Ctrl+I` (斜体), `Ctrl+K` (链接)。
- **源码折叠装饰器** (可选增强)：为避免用户手动编辑 Markdown 时因长图片占位符受到干扰，设计简单的 CodeMirror 6 Decoration 插件，将 `img://` 占位符或 base64 链接在视觉上折叠为一个“图片图标”小部件，光标点击时才展开。

### 3.4 提示横幅组件
- 在文章（公众号）渲染区顶部检测 `img://` 前缀。
- 展示提示引导用户：引导其去设置中心配置图床，或提示裂图风险。
